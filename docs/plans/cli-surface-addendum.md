# jk Project Plan Addendum: jj CLI Surface and Flag-Driven Design

> Addendum to `jk Golden Plan`.
>
> Source: `jj full CLI help export`, generated 2026-06-23T04:08:31Z from `jj 0.42.0`.
>
> Purpose: make the full `jj` command/flag surface actionable for `jk` product design, keybindings,
> command construction, testing, documentation, and roadmap decisions.

## 0. Executive summary

The full `jj` CLI surface changes the `jk` plan in one important way: `jk` cannot treat each command
as a separate button with a separate form. The CLI is organized around **reusable flag families**:

- global execution context: repository, working-copy snapshot policy, operation-time travel,
  immutability, operation integration, color/pager/noise, config overlays;
- object selectors: revsets, filesets, operations, bookmarks, tags, remotes, workspaces;
- display controls: graph/no-graph, template, limit, reversed, count, patch/no-patch;
- diff formatting controls: summary, stat, types, name-only, git, color-words, external tool,
  context, whitespace;
- source/destination roles: from, into/to, revision, source, branch, onto, insert-after,
  insert-before;
- editor/tool modes: inline message, stdin, external editor, interactive diff editor, merge tool,
  formatter/fix tool;
- safety controls: dry-run, allow-private, allow-empty-description, allow-backwards,
  keep-emptied, restore-descendants, ignore-immutable, no-integrate-operation;
- config/template languages: revsets, filesets, templates, string patterns, conditional config,
  template aliases, revset aliases.

Therefore, the project plan should treat `jk` as a **flag-family TUI**, not a command-by-command
TUI. The UI should expose common flag families as reusable overlays and role pickers, and every
workflow should still be able to fall back to `:` command mode.

The most important roadmap changes are:

1. Add a normalized `JjCommandSpec` layer before implementing more commands.
2. Add a reusable **View Options** overlay, keyed by `V`, for diff/display/template/graph options.
3. Reserve standalone `v` for `evolog` / evolution history; use `V` for view/display.
4. Add a reusable **Run Options** drawer for global options and advanced safety toggles.
5. Make workspaces (`W`) early core scope, not advanced scope.
6. Make bookmark/git push safety and conflicted bookmark resolution first-class.
7. Treat `jj util markdown-help` or equivalent generated help as a compatibility input for
   command/flag manifests.
8. Keep the core UI focused-screen based, not pane-first.

## 1. Terminology

This addendum uses these terms consistently:

- **Command fidelity**: `jk` actions can be explained as exact `jj` commands or command sequences.
- **Presentation fidelity**: `jk` honors configured jj output semantics: revsets, templates, graph
  style, colors, diff formats, node symbols, IDs, and operation/bookmark/workspace semantics.
- **Flag family**: a set of options repeated across commands with the same broad meaning.
- **Role picker**: an interaction that chooses source, destination, revision, operation, fileset,
  bookmark, or remote roles before building a command.
- **Run Options drawer**: an advanced overlay for global options such as `--ignore-working-copy`,
  `--no-integrate-operation`, `--ignore-immutable`, `--at-operation`, `--config`, and
  `--config-file`.
- **Focused screen**: a single-purpose view in the `jk` stack. This is not a persistent pane layout.

## 2. Normalized command model

Every `jk` action should eventually resolve to a normalized command spec before it renders, previews,
or runs.

```rust
struct JjCommandSpec {
    program: Program,                  // usually "jj"; sometimes external command for ! mode
    global: GlobalOptions,
    command: Vec<OsString>,            // ["rebase"], ["git", "push"], ["workspace", "add"], ...
    args: Vec<CommandArg>,
    stdin: Option<CommandStdin>,
    execution: ExecutionMode,
    refresh: RefreshPlan,
    preview: PreviewPolicy,
    provenance: CommandProvenance,     // key action, command mode, generated by wizard, etc.
}

struct GlobalOptions {
    repository: Option<PathBuf>,        // -R/--repository
    working_copy: WorkingCopyPolicy,    // default, ignore, snapshot-only later if jj adds it
    operation: OperationLoadPolicy,     // latest, --at-operation OP
    operation_integration: OperationIntegrationPolicy, // default vs --no-integrate-operation
    immutability: ImmutabilityPolicy,   // default vs --ignore-immutable
    output: OutputPolicy,               // color, quiet, no-pager
    config_overlays: Vec<ConfigOverlay>,// --config, --config-file
}

enum ExecutionMode {
    RenderReadOnly,
    ConfirmMutation,
    ConfirmExternalEditor,
    ConfirmExternalDiffTool,
    DryRunThenConfirm,
    ForegroundInteractive,
    BackgroundNever, // jk should not promise background delivery
}
```

### 2.1 Why this matters

The same command can become read-only, mutating, external-editor-driven, or dry-run-first depending
on flags:

- `jj diff` is read-only, but `jj diffedit` opens an external diff editor and mutates a revision.
- `jj split` is interactive by default when no filesets are provided, but non-interactive with
  filesets.
- `jj squash` can move all changes, only selected filesets, or open an interactive diff editor.
- `jj git push --dry-run` is read-only preview; `jj git push` mutates a remote.
- `jj --no-integrate-operation rebase ...` mutates repository state but intentionally leaves the
  result out of the integrated operation log until the user inspects or integrates it.

Without a normalized command model, these distinctions leak into one-off UI code.

## 3. Global options are product features

Every `jj` command shares global options. `jk` should not hide them entirely as transport details.

### 3.1 Repository selection: `-R`, `--repository`

Design consequence:

- `jk -R path` must carry through to every generated `jj` command.
- A future repository picker should set this once for the whole view stack.
- Command history should record the repository path or an intentionally redacted short form.
- Betamax fixtures should assert `-R` propagation in at least one workflow.

User-facing UI:

```text
:global repository /path/to/repo
```

or in a future command palette:

```text
Repository: current / choose / clear
```

### 3.2 Working-copy policy: `--ignore-working-copy`

Default `jj` commands snapshot the working copy at command start and update it at command end when
needed. `--ignore-working-copy` avoids snapshot/update and is implied by `--at-operation`.

Design consequence:

- Read-only screens that poll frequently may need an explicit stale/snapshot policy.
- `jk` should expose this as an advanced global toggle, not a normal workflow key.
- When a screen is loaded at an earlier operation, `jk` should visibly show "working copy ignored".
- If auto-refresh exists later, it should not silently switch snapshot policy.

Suggested UI:

```text
Run Options
  [ ] Ignore working copy / show stale state
```

For `--at-operation`, show as read-only time-travel context by default.

### 3.3 Operation integration: `--no-integrate-operation`

This is powerful enough to be a `jk` differentiator. It runs a command, creates an operation, but
does not integrate it into the operation log or update the working copy. The result operation ID can
be inspected with `--at-op`, restored with `op restore`, or integrated with `op integrate`.

Design consequence:

- This can become a safe preview/sandbox mode for complex mutations.
- It is not a no-op and not a dry run; side effects outside the repo can still happen.
- Never use it automatically for `git push`, `gerrit upload`, or any command with remote side effects.
- For local history mutations, `jk` can offer "simulate as operation" in the Run Options drawer.

Suggested UI:

```text
Run Options
  [ ] Do not integrate operation
      Runs jj, prints operation ID, leaves working copy unchanged.
      Safe for local repo preview; not safe for remote side effects.
```

After running:

```text
Operation created but not integrated: abc123
Enter inspect   i integrate   r restore   y copy op id   Esc discard view
```

### 3.4 Immutability override: `--ignore-immutable`

This bypasses checks against immutable commits but does not change the `immutable_heads()` revset or
the `immutable` template keyword.

Design consequence:

- This should never be a one-key action.
- Any command requiring this should show "rewrites immutable commits" in preview.
- Put under Run Options, not under ordinary command role pickers.
- Command preview should include the flag before the command, as jj global syntax.

Suggested UI:

```text
Run Options
  [ ] Allow rewriting immutable commits (--ignore-immutable)
```

### 3.5 Operation time travel: `--at-operation`, `--at-op`

This is central to `jk`'s operation-log story.

Design consequence:

- Operation log screen should support `l` = open log at operation, `s` = status at operation,
  `d` = diff/show changes at operation.
- Screens loaded with `--at-op` must be clearly labeled.
- Mutating at a prior operation is possible but should be hidden behind advanced confirmation;
  it creates a concurrent operation. Normal users expect `--at-op` to be inspect-only.

Suggested UI:

```text
Operation abc123
Enter op show   d op diff   l log at op   s status at op   r restore   v revert
```

Screen title:

```text
jk log --at-op abc123  [working copy ignored]
```

### 3.6 Output plumbing: `--color`, `--quiet`, `--no-pager`

Design consequence:

- `jk` should internally use `--no-pager`.
- If using jj-rendered output, `jk` should choose color policy explicitly.
- If using native rendering, `--color` is mostly a command-mode/output concern.
- `--quiet` may be useful for command output panes, but should not be applied blindly because
  user-visible warnings and hints matter.

### 3.7 Config overlays: `--config`, `--config-file`

Design consequence:

- `jk` command mode must preserve raw config flags and not over-parse TOML values incorrectly.
- Native rendering must account for conditional config by command and workspace.
- Test fixtures should use `JJ_CONFIG=` or `--config-file` to make deterministic output.
- A future "config profile" feature should emit command previews using `--config-file`, not private
  internal state.

Suggested UI:

```text
Run Options
  Config overlays
    + --config NAME=VALUE
    + --config-file PATH
```

Critical implementation note: if `jk` natively renders a command equivalent to `jj log`, it must
load config in a way equivalent to the command context `log`, because jj supports conditional config
scoped by command.

Do not implement each command's selector UI from scratch. Build these once.

Used by:


- operation log screen is the selector;
- operations can be selected by full ID or unambiguous prefix;
- show operation metadata, args, duration, user/host/time;
- map selected op to `--at-op` screen contexts;

### 4.4 Bookmark/tag/remotes/workspace selectors

Bookmark selector is not just names; it includes local, remote, tracked, untracked, conflicted,

selector chooses names and paths.

These selectors deserve focused screens:

```text

```

The exact key for a future Git/remotes screen can be revisited. `F` and `P` remain fetch/push entry
points.


- Template editing/completion is not a log-only feature.
- Native rendering must either evaluate jj templates or explicitly route to jj-rendered output for

- A template error should degrade into a clear error panel with the attempted command.

## 7. Read-only and inspection command mapping

### 7.1 `log`

Flags that matter:

```text
-r/--revision
-n/--limit
--reversed
-G/--no-graph
-T/--template
-p/--patch
--count
diff formatting options
FILESETS
```

Design:

- Default graph screen maps to configured log revset/template.
- `:` command mode should support exact `log` args.
- Revset editor should be first-class and completion-capable.
- File filters should use the shared fileset selector.
- `log -p` should be a valid view mode, but not the primary default.
- `--count` can be an output/command result, not an interactive screen.

### 7.2 `show`

Flags:

```text
REVSETS
-T/--template
--no-patch
diff formatting options
```

Design:

- `Enter` on a revision opens `show`.
- `V` controls patch/no-patch and diff formatting.
- Multiple marks can become `jj show A B C`.
- If native details exist, they must honor template or clearly say "structured view".

### 7.3 `diff`

Flags:

```text
-r/--revisions
-f/--from
-t/--to
FILESETS
diff formatting options
```

Design:

- no marks + cursor => `jj diff -r <cursor>`;
- one mark + cursor => `jj diff --from <mark> --to <cursor>`;
- two ordered marks => `jj diff --from <first> --to <second>`;
- more than two marks => prompt: "combined revisions diff" (`-r`) vs choose from/to;
- marked files append filesets;
- `S` sets `--stat`.

### 7.4 `interdiff`

Flags:

```text
--from
--to
FILESETS
diff formatting options
```

Design:

- This is not the same as `diff --from A --to B`.
- Add `I` or `V i` later; do not overload `d`.
- Best home: from evolog screen or from two versions of the same change.
- Useful for "what changed since push" workflows.

### 7.5 `evolog`

Flags:

```text
-r/--revisions
-n/--limit
--reversed
-G/--no-graph
-T/--template
-p/--patch
diff formatting options
```

Design:

- standalone `v` opens evolution log for selected change.
- From evolog, `Enter` show version, `d` diff version, `I` interdiff between marked versions.
- `p`/patch should live under View Options.
- Useful early because `jk` is change-ID oriented and jj changes evolve.

### 7.6 `status`

Flags:

```text
FILESETS
```

Design:

- `s` opens status.
- Status should surface working-copy commit, parents, working-copy change summary, file conflicts,
  and conflicted bookmarks.
- From status, files can be marked for commit/restore/squash/split.
- Conflicted bookmarks should offer a jump to Bookmarks conflict resolution.

### 7.7 `file`

Subcommands:

```text
annotate
chmod
list
search
show
track
untrack
```

Design:

- `file list` can power file pickers.
- `file show` can power file content detail screens.
- `file annotate` is a later focused screen (`A` in file view, or action menu).
- `file chmod` is a file action.
- `file track`/`untrack` belong in status/file action menu.
- `file search` can become repository search later; start with command mode.

## 8. History editing command mapping

### 8.1 `describe` and `metaedit`

`describe` flags:



Design:

- `m` = inline message prompt => `jj describe -m`.
- `M` = external editor => `jj describe`.
- `a m` = metaedit menu.
- Use stdin for multi-line message prompts to avoid shell quoting.
- `--editor` after a message should be offered as "edit before saving".
- `metaedit --update-change-id` and author rewrites require confirmation.

### 8.2 `new`, `commit`, `edit`, `prev`, `next`

`new` flags:


```text

```

`edit`:

```text

```

`prev`/`next`:


- `n` = `jj new <cursor-or-marked-parents>`.
- `N` = new with message.
- `c` = commit current working-copy change; only direct in status/@ context.
- `e` = edit selected revision; preview should mention jj recommends `new` + `squash` in many cases.
- `prev`/`next` are working-copy movement commands, not UI navigation. Keep out of `j/k`.
- Add `:prev`, `:next` support via command mode immediately; later add action menu entries.

### 8.3 `rebase`

Important flags:

```text

-s/--source
-r/--revision
-o/--onto
-A/--insert-after
-B/--insert-before
--skip-emptied
--keep-divergent
--simplify-parents
```

Design:

- `R` always opens the rebase wizard or preview; it should not run blindly.
- Source mode defaults to `branch` for ordinary marked stack workflows.
- Destination mode defaults to `onto`.
- Expose `after`/`before` in the destination picker.
- Support repeated destinations for merge parent creation.
- Advanced toggles:
  - skip emptied;
  - keep divergent;
  - simplify parents;
  - ignore immutable.
- Rebase preview should eventually show ghost graph changes.
- The command preview must use canonical `-o`; `-d` remains accepted in command mode only.


Design:

- All live under action menu.
- `abandon` confirmation must mention descendant rebase and bookmark behavior.
- `abandon --retain-bookmarks` and `--restore-descendants` belong in advanced options.
- `revert` uses the same destination role picker as rebase but creates inverse changes.
- `duplicate` uses the same destination role picker as `new`/`rebase`.
- `parallelize` and `simplify-parents` should be preview-first because their effects are graph-only
  and not obvious from command names.

### 8.6 `resolve`

Flags:

```text
FILESETS
-r/--revision
-l/--list
--tool
```

Design:

- Conflicted rows should show conflict affordance.
- `a R` opens resolve flow.
- Start with `jj resolve --list` as read-only conflict picker.
- Run `jj resolve` foreground with terminal restored.
- Offer built-in `:ours`/`:theirs` in advanced picker only with very clear wording.

### 8.7 `sign` and `unsign`

Design:

- Later action menu entries.
- Signing may require hardware devices; it is foreground/confirmation work.
- Do not run automatically.
- Show configured key if discoverable.

### 8.8 `arrange` and `bisect`

Design:

- `arrange` is interactive graph arrangement. It may overlap with `jk`'s future native graph preview.
  Keep in command mode until there is a clear integration story.
- `bisect run` is command-execution driven. `jk` can later provide a bisect wizard, but `!` and `:`
  command modes cover it initially.

## 9. Operation log command mapping

Subcommands:


```text
-n/--limit
--reversed
-G/--no-graph

-p/--patch
--no-op-diff

--show-changes-in
```

Mutation flags:


- `o` opens op log.
- `u`/`U` are undo/redo global.
- Operation restore/revert are confirmed.
- Operation abandon is destructive history cleanup and should be hidden under operation action menu,
  with strong warning.
- Operation integrate pairs with `--no-integrate-operation`; it becomes important if jk offers
  sandboxed local mutation previews.
- `--show-changes-in` should use revset selector and live in View Options on operation screens.

## 10. Bookmarks, tags, remotes, and Git push/fetch

### 10.1 Bookmark screen

Subcommands:

```text
advance
create
delete
forget
list
move
rename
set
track
untrack
```

Important flags:

```text
bookmark list:
  --all-remotes
  --remote
  --tracked
  --conflicted
  --revision
  --template
  --sort

bookmark move:
  --from
  --to
  --allow-backwards

bookmark set:
  --revision/--to
  --allow-backwards

bookmark forget:
  --include-remotes

bookmark track/untrack:
  --remote
```

Design:

- `B` opens bookmarks screen.
- Bookmark screen needs filters:
  - local;
  - all remotes;
  - tracked;
  - conflicted;
  - by remote;
  - by target revision.
- Conflicted bookmarks are first-class, not just errors.
- `bookmark delete` and `bookmark forget` are different:
  - delete propagates deletion to remotes on push;
  - forget only stops local tracking/knowledge.
- Moving backwards/sideways requires an explicit toggle and warning.
- `bookmark advance` deserves a quick action for "advance nearest bookmark to selected revision".

### 10.2 Tags

Subcommands:

```text
tag list
tag set
tag delete
```

Design:

- `T` opens tags screen.
- Tag list likely mirrors bookmark list but smaller.
- Tag deletion must clarify it does not abandon revisions.

### 10.3 Git fetch

Flags:

```text
--branch
--tracked
--remote
--all-remotes
```

Design:

- `F` opens fetch screen or runs default fetch if unambiguous and user config says so.
- If multiple remotes exist, choose remote.
- Expose tracked/all-remotes/branch filters.
- Fetch may abandon local commits considered abandoned by remote depending on config; show a note in
  advanced docs and command output.

### 10.4 Git push

Flags:

```text
--remote
--bookmark
--all
--tracked
--deleted
--allow-empty-description
--allow-private
--revision
--change
--named
--dry-run
--option
```

Design:

- `P` always starts with dry-run preview.
- Push screen should explain that remote is selected with `--remote`; jj does not derive it from
  tracked bookmarks.
- Offer modes:
  - current selected bookmark;
  - all bookmarks;
  - tracked bookmarks;
  - deleted bookmarks;
  - generated bookmark for selected change;
  - named bookmark.
- Safety checks and conflicted bookmarks should drive remediation UI:
  - fetch remote;
  - resolve bookmark conflict;
  - track bookmark;
  - allow-private/allow-empty-description only as explicit advanced toggles.
- Never use `--no-integrate-operation` as a "safe push preview"; it does not prevent remote side
  effects.

### 10.5 Git remote management

Subcommands:

```text
remote add
remote list
remote remove
remote rename
remote set-url
```

Design:

- Remote management can live under Git screen later.
- Add/set-url/remove need confirmation.
- Fetch/push screens should link to remote management when no remote exists.

### 10.6 Git init/clone/colocation/import/export/root

Design:

- `git clone` and `git init` are onboarding flows, not core in-repo workflows.
- `git colocation status/enable/disable` is advanced and should be command-mode first.
- `git import/export` are automatic in colocated workspaces; expose in command mode and Git screen.
- `git root` is diagnostic.

### 10.7 Gerrit

Design:

- `gerrit upload` is important for Gerrit users but a specialized workflow.
- It has many review-specific flags: remote branch, remote, dry-run, reviewers, cc, labels, topic,
  hashtags, message, edit, WIP/ready/private/remove-private, notify, submit, validation, custom,
  trace, deadline.
- Treat as a later publish provider screen, parallel to push, with dry-run-first support.
- Keep `:gerrit upload` as the initial support.

## 11. Workspaces are early core scope

The help export reinforces that workspaces are a primary jj concept:

```text
workspace add
workspace forget
workspace list
workspace rename
workspace root
workspace update-stale
```

Important semantics:

- Workspaces are additional working copies attached to the same repo.
- Each workspace has its own working-copy commit.
- Multiple workspaces are indicated in log as `<workspace name>@`.
- Each workspace has its own sparse patterns.
- `workspace add` can choose destination, name, parent revisions, message, and sparse pattern policy.
- `workspace forget` does not touch files on disk.
- Stale workspaces can be updated.

Design:

- `W` opens Workspaces screen in the 0.5/0.6 horizon, not later.
- Workspaces screen is focused, not a permanent sidebar.
- Support:
  - list workspaces;
  - identify current workspace;
  - add workspace;
  - rename current workspace;
  - forget workspace with explicit "files remain on disk" note;
  - update stale workspace;
  - root/path display;
  - status/diff/log for selected workspace;
  - sparse pattern mode at creation.

Workspace add wizard:

```text
Destination: ../repo.feature
Name: feature
Parents: selected revision(s) / current parents / typed revset
Message: optional
Sparse patterns: copy / full / empty
Preview: jj workspace add --name feature -r abc123 -m "..." --sparse-patterns copy ../repo.feature
```

## 12. Sparse checkout and filesystem scale

Sparse subcommands:

```text
sparse edit
sparse list
sparse reset
sparse set --add --remove --clear
```

Design:

- Sparse patterns matter most with workspaces.
- Put sparse controls in Workspaces screen `V`/actions or a later Sparse screen.
- `sparse edit` is an external editor flow.
- `sparse set` needs a pattern editor and preview.

Scale/config notes from help/config:

- fsmonitor/watchman settings can affect snapshot performance in large repos.
- snapshot auto-track and max-new-file-size influence status/file tracking.
- `jk` should document performance recommendations, but not configure them silently.

### 16.1 Store a versioned help fixture

- Keep at least one fixture for the minimum supported jj and one for latest tested jj once release
  cadence stabilizes.

## 19. Codex decision rules for future work

When Codex implements a feature:

1. Identify the jj command path and all relevant flag families.
2. Add or update `JjCommandSpec` tests before UI code.
3. Decide whether the workflow is:
   - read-only render;
   - confirmed mutation;
   - external editor/tool;
   - dry-run then confirm;
   - command-mode only.
4. Use existing selectors and role pickers; do not create one-off prompts.
5. Update generated help/hotbar entries.
6. Add a Betamax validation tape.
7. Add docs only if the workflow is user-visible.
8. Keep command preview canonical and jj-shaped.
9. If a flag is experimental in jj, mark it experimental in the UI.
10. If a command has remote side effects, never use local operation sandboxing as a dry-run substitute.

## 20. Open questions

These should be resolved by implementation experiments:

- Should `jk` expose `--no-integrate-operation` as a public "sandbox mutation" feature in 1.0, or
  keep it advanced/hidden until operation integrate/restore UX is excellent?
- Should `interdiff` get direct `I`, or live under evolog and View Options?
- Should there be a dedicated Git/remotes screen key, or are `F`, `P`, `B`, and command mode enough?
- How much of jj's template language should `jk` natively evaluate before falling back to jj-rendered
  output?
- Should `prev`/`next` receive direct keys, or remain command-mode/action-menu features because they
  mutate working-copy state?
- Should `arrange` be delegated to jj foreground mode or eventually integrated into native graph
  editing?
- How should `jk` represent multiple workspaces in graph rows when several `<workspace>@` markers
  exist?

## 21. Bottom line

The complete jj CLI surface does not invalidate the golden plan. It sharpens it.

`jk` should become a jj-native TUI by building reusable UI primitives for jj's reusable flag
families:

```text
selectors + role pickers + view options + run options + command preview + command mode
```

That is the only scalable way to cover "most of jj" without producing a giant, inconsistent keymap.
