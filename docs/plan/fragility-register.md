# Fragility Register

This register tracks places where `jk` depends on soft agreements with `jj`: rendered output shapes,
underspecified command behavior, inferred state, or duplicated logic. These assumptions are allowed
when they are narrow and useful, but they should be visible during planning and review.

## Current Code

  | Area                      | Current dependency                                                                                                             | Risk                                                                                | Mitigation                                                                                                                                                                                                        |
  | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
  | Render pipeline           | stdout -> ANSI parser -> styled spans -> Ratatui items                                                                         | Each conversion can lose semantic structure or style intent                         | Keep conversion local; prefer shared rendering APIs if structure expands                                                                                                                                          |
  | Semantic inference        | Meaning reconstructed from rendered text after `jj` had it                                                                     | `jk` can misread or discard meaning that code paths could preserve                  | Prefer structured output, templates, `jj_cli`, `jj_lib`, or upstream APIs                                                                                                                                         |
  | View decisions            | Row composition, templates, colors, graph, and labels                                                                          | `jk` can diverge from jj defaults or user configuration                             | Prefer APIs that expose semantic data and renderable view parts together                                                                                                                                          |
  | Log row expansion         | Compact row re-parsed before richer inline detail is attached                                                                  | Expanded rows can disagree with the compact row or lose identity                    | Prefer a row contract with ids, graph info, styled spans, and detail hooks                                                                                                                                        |
  | Action selection          | Mutation inputs inferred from selected rendered rows                                                                           | Actions can target the wrong revisions or lose multi-row meaning                    | Carry exact revision ids and graph relationships through guided flows                                                                                                                                             |
  | Rebase preview flow       | Explicit source change ids, current-row destination, and `jj rebase -r ... -o ...` semantics                                   | Command-summary wording can be mistaken for a simulated before/after graph          | Keep roles explicit, build the exact command, state `-r`/`-o` without graph simulation, reveal a rebased source, and show both review paths                                                                       |
  | Squash preview flow       | Explicit source change ids, current-row destination, and `jj squash --from ... --into ... --use-destination-message` semantics | Source/destination role inference or description behavior can drift from jj CLI     | Preserve source/destination roles through the prompt and preview, keep `--use-destination-message` visible, reveal the destination after refresh, and show `jj undo`                                              |
  | Absorb preview flow       | Current graph-row source, selected candidate destinations, and `jj absorb` line-placement semantics                            | Line placement, ancestry, no dry-run, source empty/abandon, op-show opaque          | Use exact ids only; do not infer ancestry; preview caveats; preserve `jj undo` and rendered `jj op show -p`                                                                                                       |
  | New from parents          | Exact graph-row change ids are passed as positional `jj new` parent revsets                                                    | Selected-parent ordering or revset resolution can create an unexpected merge        | Keep parents sourced only from exact graph metadata, sort explicit selections by current graph row order, preview every parent id, and show `jj undo` after run                                                   |
  | Working-copy navigation   | Exact-row `jj edit` and `jj next --edit` / `jj prev --edit` movement relative to current `@`                                   | Bare `next`/`prev` can drift with `ui.movement.edit`; topology can be ambiguous     | Always use `exactly(change_id(...), 1)` for `edit`; always pass `--edit` to `next`/`prev`; preview that `next`/`prev` ignore the highlighted row; reveal current `@`; preserve raw ambiguity output and `jj undo` |
  | Describe metadata         | Graph describe targets come from exact row change ids, while status describe targets delegate to `@`                           | A selected graph row could be confused with `@`, or `@` could drift before run      | Use exact change-id revsets for graph targets, keep `@` visible for status/current working-copy targets, preview the target and message, and show `jj undo`                                                       |
  | Commit working copy       | `jj commit --message <message>` always acts on `@` and creates a new working-copy change on top                                | Launching from graph can imply the selected row is committed                        | Preview and help state that commit targets `@`, never pass a selected revision, keep the new-working-copy effect visible, and show `jj undo` after success                                                        |
  | Abandon exact target      | Single graph-row abandon carries row metadata and executes `exactly(change_id(...), 1)`                                        | Ambiguous rows, source selections, or rendered labels could target wrong changes    | Expose abandon only for a single exact current revision and carry the id through the follow-up instead of reparsing labels                                                                                        |
  | Abandon emptiness         | `jj diff -r exactly(change_id(...), 1) --summary` stdout decides confirmation tier                                             | Diff summary output or the preview-to-run window can alter the safe tier            | Treat only trimmed-empty stdout as empty, recheck before empty-preview execution, require typed confirmation after non-empty drift, and preserve failures readably                                                |
  | Abandon title             | A narrow `description.first_line()` template supplies the preview title                                                        | Template behavior can drift or fail for unusual descriptions                        | Keep the title non-authoritative; exact id and diff summary remain the confirmation contract                                                                                                                      |
  | Restore exact fileset     | Path restore builds `root-file:"<path>"` from owned file-view path state and `jj` string-literal semantics                     | Incorrect quoting could restore the wrong file or fail on spaces and metacharacters | Build the fileset centrally, use exact owned paths only, and cover spaces, quotes, backslashes, and fileset metacharacters in command-construction tests                                                          |
  | Status exact paths        | Rendered `jj status` rows are parsed only for clean `M`, `A`, and `D` single-path rows                                         | Future or configured status output could make a rendered line ambiguous             | Keep the parser narrow, disable headers, renames, conflicts, untracked-looking and multi-path rows, test degradation, and prefer structured status data if actions expand                                         |
  | Restore rewrite semantics | `jj restore --changes-in exactly(change_id(...), 1)` can rewrite the target change and rebase descendants                      | Users could read restore as an isolated patch removal instead of a history rewrite  | Keep the exact target visible, describe restore as removing the selected forward diff from that revision, preserve raw jj output, and keep `jj undo` visible                                                      |
  | Revert path scope         | Installed `jj 0.41.0` exposes `jj revert -r <rev> -o <rev>` but no path/fileset arguments                                      | Advertising path-scoped revert would imply command support that does not exist      | Expose revert only as whole-revision reverse application into `@`; keep path-scoped revert out of menus, previews, and tests                                                                                      |
  | Restore/revert previews   | Guided previews render the forward `jj diff` for the selected revision or exact path instead of a dry run or graph simulation  | Users could mistake the preview for the final post-command graph or file state      | Label the diff as preview input, state that restore removes or revert reverse-applies that forward diff, and keep `jj undo` as the recovery path                                                                  |
  | Push targeting            | Status/bookmark/graph push flows select an explicit bookmark, revision, or remote; single-remote pushes skip remote selection  | Status-context default targeting can drift if `jj` changes push resolution          | Keep the selected target visible in the prompt, preview, and result context; prefer structured target state                                                                                                       |
  | Push preview output       | Push preview and completion screens render command labels and `jj git push` result text                                        | Output wording or shape changes can hide or rename success/failure details          | Preserve raw CLI text, keep command-construction tests, and add parsing only if the flow grows                                                                                                                    |
  | Fetch remote targeting    | Explicit remote fetch uses remote names from `jj git remote list` and passes `--remote exact:<name>` to `jj git fetch`         | `--remote` string-pattern semantics or remote-list output could drift               | Keep default `jj git fetch` untouched, show the selected remote and exact pattern in preview/result context, test argv construction, and prefer structured remote data if sync flows expand                       |
  | Log selection state       | Exact change-id multiselect state kept separately from row index                                                               | Reordered rows and filtered views can still invalidate cached selected rows         | Store selected items by exact id, reconcile against current view on refresh/mode changes                                                                                                                          |
  | Graph grouping            | Rendered `jj log` lines and graph glyph layout                                                                                 | Output shape changes can affect row selection or target extraction                  | Keep the parser narrow; test elided, malformed, and non-revision rows                                                                                                                                             |
  | Revision identity         | Template output for change and commit ids                                                                                      | Template behavior or fields may change                                              | Keep templates minimal; move to typed API if identity extraction expands                                                                                                                                          |
  | Direct startup views      | Local inference of `show` and `diff` targets from startup args                                                                 | Option edge cases can be misread                                                    | Preserve raw args; test supported direct-view cases only                                                                                                                                                          |
  | File headings             | Default and `--git` rendered diff/show headings                                                                                | Heading changes can break sticky file context                                       | Preserve raw document lines; test default and git diff heading forms                                                                                                                                              |
  | Sticky projections        | Active file inferred from scroll position and headings                                                                         | Unusual output can produce no active file                                           | Degrade to a plain scrollable document                                                                                                                                                                            |
  | Diff format state         | Detecting and adding `--git` format arguments                                                                                  | Other diff tools or future formats may not match assumptions                        | Keep behavior explicit and local to diff/show command construction                                                                                                                                                |
  | Operation-log rows        | Rendered `jj operation log` item starts and graph-only lines                                                                   | Output shape changes can split or merge visible operations                          | Keep parsing narrow; test multi-line rows, ANSI spans, and graph-only rows                                                                                                                                        |
  | Operation-log ids         | Separate `self.id()` template output paired by row order                                                                       | Rendered rows and metadata rows can drift or differ under new args                  | Keep both calls on `--at-op=@ operation log`; degrade to non-copyable rows                                                                                                                                        |
  | Operation detail ids      | Selected operation ids are passed to `jj operation show <id>` and `jj operation diff --operation <id>`                         | `jj` could change operation detail command shape or option names                    | Keep command-construction tests around both invocations and keep disabled behavior when an operation-log row has no id                                                                                            |
  | Operation detail output   | Rendered `jj operation show` and `jj operation diff` output is preserved as a plain document                                   | Output wording and transaction structure can change without notice                  | Do not parse operation detail semantics; keep scroll/search/copy over styled rendered lines only                                                                                                                  |
  | Operation undo/redo       | `jj undo` and `jj redo` operate on the current repo's global operation cursor, not a selected row                              | Launching from operation log can imply the selected operation id is targeted        | Preview, help, status context, and command-construction tests state that no operation id is passed; unavailable redo is preserved as raw jj error output                                                          |
  | Operation restore/revert  | `jj operation restore <id>` and `jj operation revert <id>` use the exact selected operation-log id                             | Restore/revert recovery could be confused with global undo/redo or row labels       | Expose the actions only from rows with paired operation ids, preview the exact id and command, require confirmation, preserve raw result/error output, and keep `jj undo` visible                                 |
  | Bookmark metadata         | Separate bookmark template output emits name, remote, and target ids, then pairs one metadata row to each rendered row         | Rendered rows and metadata rows can drift or differ under new args                  | Treat missing metadata as nonlocal; mark local only when the machine `remote` field is empty; keep row-order parser tests around remote-exposed output                                                            |
  | Bookmark mutation target  | Local create/set/move flows use exact graph change ids or status `@`                                                           | Status `@` can move between preview and execution                                   | Keep `@` visible in status previews, use exact `change_id()` revsets for graph targets, require confirmation, and show `jj undo` after execution                                                                  |
  | Bookmark exact patterns   | Move/delete use jj string patterns such as `exact:"feature/name"`                                                              | Incorrect escaping could match the wrong bookmark or fail unexpectedly              | Build exact patterns centrally with string-literal escaping and cover quote/backslash names in command-construction tests                                                                                         |
  | Bookmark delete semantics | `jj bookmark delete` is destructive and distinct from `forget`                                                                 | Users could confuse local deletion with remote-tracking forget/untrack behavior     | Preview says delete, not forget; require ActionOutput confirmation; keep undo visible                                                                                                                             |
  | Bookmark tracking state   | Track/untrack are disabled because exact remote/tracking metadata is absent                                                    | Rendered labels such as `@origin` or `main@origin` could be misread as state        | Do not infer tracking from rendered labels; expose tracking only after `BookmarkItem` carries explicit metadata                                                                                                   |
  | Bookmark list filtering   | Default bookmark list output can include or hide remote rows depending on jj behavior and args                                 | A selected row may not represent a local mutable bookmark                           | Delete requires a selected row whose paired metadata proves an empty remote field; do not infer localness from rendered labels or indentation                                                                     |
  | File inspection           | Rendered `jj file list` rows and `jj file show` document output                                                                | Label drift or heading shape changes can break copy, refresh, or sticky anchors     | Keep exact paths separate from labels; preserve raw document lines; test path-preserving refresh                                                                                                                  |
  | Resolve conflict list     | `jj 0.41` template methods `self.conflicted_files()`, `entry.path()`, `entry.file_type()`, and `entry.conflict_side_count()`   | Template names, field semantics, or conflict-path exactness can drift               | Keep the contract narrow, parse one JSON object per line with safe degradation, preserve exact paths separately from row text, and revisit typed APIs before writes                                               |

## Planned Screens

Each planned screen should add entries here before implementation if it needs output parsing or
duplicated jj semantics.

Narrow machine-oriented templates are different from arbitrary user-facing rendered output. They are
acceptable when they expose explicit fields for `jk`; they should not become a second copy of jj's
template language or presentation model.

  | Screen or workflow | Expected dependency                                            | Preferred contract                                                                                                                   |
  | ------------------ | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
  | Status             | Rendered `jj status` sections, working-copy file groups        | Rendered output first; structured data if actions need precise file sets                                                             |
  | Operation log      | Rendered `jj op log` graph and operation ids                   | Narrow parser for operation ids; use `--at-op=@` to avoid view-time snapshots; typed API if recovery flows need transaction detail   |
  | Bookmarks          | Rendered bookmark lists and local/remote tracking markers      | Rendered output first; narrow local bookmark metadata now, and `jj_lib` or structured output if actions need exact tracking state    |
  | File list / show   | File list rows and file document output                        | Structured output or narrow templates once file actions need mutation-capable exact-path semantics                                   |
  | Resolve            | Conflict listing and file state                                | Narrow `self.conflicted_files()` template now; typed API preferred before guided conflict resolution or exact fileset writes expand  |
  | Rewrite flows      | Preview output, action menu role prompts, and command planning | `jj` command previews can drift from direct shell inference; avoid executing rewrites without explicit preview and role confirmation |
  | Sync flows         | Fetch/push stderr/stdout and bookmark state                    | Prefer command passthrough until structured state is needed                                                                          |

## Duplication Watchlist

These areas are especially likely to drift if copied into `jk`:

- template parsing, rendering, and formatting;
- config interpretation, aliases, and style/color resolution;
- revset parsing and evaluation;
- fileset parsing and evaluation;
- graph layout, edge rendering, and glyph semantics;
- default row composition and user-configured view choices;
- semantic row fields paired with renderable styled segments;
- command planning, defaults, and safety checks;
- operation transaction behavior;
- conflict and merge state modeling;
- bookmark and remote-tracking semantics.

The adjacent `../jj` source currently exposes promising modules for several watchlist items:
`jj_cli::templater`, `jj_cli::commit_templater`, `jj_cli::formatter`, `jj_cli::graphlog`,
`jj_cli::revset_util`, and `jj_lib`'s repo/revset/graph/diff/transaction modules. Treat those as
preferred spike targets before copying behavior locally. Also treat them as provisional until the
spike proves the public surface is usable without depending on accidental internals.

If one of these areas becomes necessary, record the chosen path in the relevant screen or workflow
plan:

- use rendered output as-is;
- add a narrow parser;
- use a purpose-built template or structured output;
- use shared rendering, config, or template APIs;
- integrate through `jj_cli` or `jj_lib`;
- request or prototype an upstream API;
- accept temporary duplication with tests and an explicit removal path.

## Build-Break Preference

Prefer integration points that fail loudly when `jj` changes:

- compile errors from typed Rust APIs;
- schema or structured-output failures;
- snapshot diffs for rendered output that is intentionally parsed;
- focused parser tests around every soft agreement;
- explicit degraded behavior when parsing fails.

Avoid behavior that silently reconstructs a different repository model from the one `jj` would show
the user.
